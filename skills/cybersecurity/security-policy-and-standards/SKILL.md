# Security Policy and Standards

## Overview

Policies state requirements; standards specify how. Effective security documents are clear enough to implement and test — not generic statements copied from the internet.

## When to Use

- Building a security policy hierarchy
- Publishing hardening baselines (CIS-aligned or custom)
- Aligning policy with compliance obligations
- Reducing ambiguity in security requirements for engineers

## Core Practices

- Separate policy (what/why) from standards (how) from procedures
- Write testable requirements
- Assign owners and review cycles
- Map controls to implementation evidence
- Involve implementers in drafting to ensure feasibility
- Version and communicate changes

## Principles

- Unenforceable policy trains people to ignore security docs
- Exceptions need process, not silent noncompliance
- Shorter, clearer documents get higher compliance
- Standards should enable automation and measurement

## Verification

- [ ] Critical domains have current policy/standard coverage
- [ ] Requirements are testable
- [ ] Exception handling is defined
