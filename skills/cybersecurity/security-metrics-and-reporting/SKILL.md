# Security Metrics and Reporting

## Overview

Security metrics should change behavior and resource allocation. Vanity counts of “alerts closed” without quality or risk context mislead leaders.

## When to Use

- Security program scorecards
- Board and executive risk reporting
- Team operational KPIs
- Demonstrating control effectiveness over time

## Core Practices

- Prefer outcome and coverage metrics over pure activity counts
- Track MTTD/MTTR, vuln age, patch SLA performance, MFA coverage, critical exposure
- Segment by asset criticality
- Pair metrics with narrative risk context
- Avoid overwhelming leaders with unprioritized red lists
- Review metric definitions for Goodhart distortions

## Principles

- What gets measured gets gamed — design carefully
- Trends matter more than single-point snapshots
- Metrics without owners do not improve
- Honest uncertainty beats false precision in risk reporting

## Verification

- [ ] Metrics map to program goals and risks
- [ ] Definitions are stable and documented
- [ ] Reports drive decisions or resource changes
