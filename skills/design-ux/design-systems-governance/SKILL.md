# Design Systems Governance

## Overview

A design system without governance drifts or becomes a bottleneck. Governance defines how changes are proposed, reviewed, versioned, and adopted.

## When to Use

- Growing from a library to a multi-team system
- Conflicts over one-off components vs system additions
- Planning release and communication cadence
- Defining who decides changes to foundational tokens or patterns

## Core Practices

- Define contribution paths (propose → review → accept)
- Set quality and accessibility bars for new components
- Version the system and communicate breaking changes
- Clarify support model and response expectations
- Measure adoption and pain points
- Create a lightweight RFC or proposal process

## Principles

- Enable teams; don’t become the department of no
- Prefer extension patterns over forbids when possible
- Document decisions so the system’s “why” survives team changes
- Governance should scale with organizational complexity

## Verification

- [ ] Teams know how to request or contribute changes
- [ ] Breaking changes are communicated with migration paths
- [ ] Adoption metrics and feedback loops exist
