# Analytics Problem Framing

## Overview

Most failed analytics work fails at the question, not the model. Framing turns vague asks into decision-oriented problems with measurable success criteria.

## When to Use

- New analytics or DS project intake
- Stakeholder requests that start as “can we look at the data…”
- Prioritizing a backlog of analysis ideas
- Defining success before building dashboards or models

## Core Practices

- Identify the decision the work should inform
- Write precise questions or hypotheses
- Define success metrics and acceptable uncertainty
- List data, time, and capability constraints
- Agree what will *not* be answered
- Choose method class only after the question is clear

## Principles

- Analysis without a decision context is expensive curiosity
- Perfect data is rare — frame for available data honestly
- Stakeholders often need options, not only a single number
- Reframe mid-project when the real question changes

## Verification

- [ ] Decision and question are explicit
- [ ] Success criteria are agreed
- [ ] Scope exclusions are stated
